# ADR-004: Team Member Identity — Local SQLite Extended with Supabase Auth
**Date:** 2026-06-04
**Status:** Accepted
**Deciders:** Boss (Eytan)

---

## Context

The app is transitioning from single-user (owner-only) to multi-user access. Team members need:

- Email-based authentication (magic link OTP via Supabase Auth)
- Profile photos
- Role-based access control (owner / manager / member)
- Invite codes so the owner can onboard staff without creating accounts on their behalf

The existing `StaffMember` type (defined in `types/index.ts`) carries:

```ts
interface StaffMember {
  id: string;
  name: string;
  role: string;          // free-text job title, e.g. "Housekeeper"
  color: string;
  voice_sample_uri?: string;
  avatar_initials: string;
}
```

There is no concept of authentication, identity provider linkage, role hierarchy, or invite flow in the current model. Two broad approaches were considered:

1. **Replace** the `staff` table with a Supabase-managed remote table and treat Supabase Auth as the single source of truth for everything.
2. **Extend** the local SQLite `staff` table with auth-related columns and introduce Supabase Auth as a complementary identity layer, preserving the existing offline-first, SQLite-primary sync strategy.

The app's data architecture is hybrid: SQLite is the primary on-device store, with Supabase as the sync target. Replacing the local table with a remote-first model would break offline capability and require a larger, riskier migration. Extending the existing table is additive and backward-compatible.

---

## Decision

**The local SQLite `staff` table is extended — not replaced — with five new columns.**

Supabase Auth is the identity provider and the source of truth for authentication. The local `staff` table remains the source of truth for team membership and role within the org.

### New columns on the `staff` table

| Column | Type | Default | Nullable | Purpose |
|---|---|---|---|---|
| `email` | `TEXT` | — | YES | Used for invite delivery and OTP pre-fill |
| `role_level` | `TEXT` | `'member'` | NO | Hierarchical role: `'owner'`, `'manager'`, or `'member'` |
| `invite_code` | `TEXT` | — | YES | UUID v4, generated when the owner first saves the profile |
| `supabase_user_id` | `TEXT` | — | YES | Populated after first OTP verification; NULL until the member authenticates |
| `avatar_url` | `TEXT` | — | YES | Supabase Storage URL or local file URI |

### Role hierarchy

```
owner > manager > member
```

`role_level` is distinct from the existing free-text `role` column. `role` continues to hold the job title (e.g. "Housekeeper"). `role_level` encodes the permission tier.

### Invite link format

```
Web:      https://[TEAM_APP_DOMAIN]/invite/[invite_code]
Deep link: meetai://invite/[invite_code]
```

`invite_code` is a UUID v4 generated by the app at owner profile save time. It is permanent and re-usable for re-authentication, but some features require a full OTP login (i.e. `supabase_user_id` must be non-NULL).

### Owner profile bootstrapping

The owner is a `staff` row with `role_level = 'owner'`. This row is created on first app launch. The owner requires no invite code. The owner must set their own email in Settings before the invite system becomes operational — staff rows without `email` cannot be invited.

### Redemption flow (summary)

1. Owner shares invite link containing `invite_code`.
2. App resolves `invite_code` against the Supabase `profiles` table to retrieve the associated email.
3. Email is pre-filled; Supabase OTP is sent.
4. On successful OTP verification, `supabase_user_id` is written back to the local `staff` row via sync.

---

## Two-State Membership Model

A staff row exists in one of two states:

```
┌─────────────────────────────────────────────────────────────────┐
│  STATE 1 — Invited (pre-auth)                                   │
│                                                                 │
│  invite_code     : set (UUID v4)                                │
│  supabase_user_id: NULL                                         │
│                                                                 │
│  Person has a link but has not completed OTP login.             │
│  Can be identified in the app by name/color.                    │
│  Cannot perform actions gated on full authentication.           │
└─────────────────────────────────────────────────────────────────┘
                          │
                          │  OTP verified
                          ▼
┌─────────────────────────────────────────────────────────────────┐
│  STATE 2 — Authenticated                                        │
│                                                                 │
│  invite_code     : retained (re-usable for re-auth)             │
│  supabase_user_id: populated (Supabase Auth UUID)               │
│                                                                 │
│  Full identity established. Auth-gated features unlocked.       │
│  Supabase Auth is authoritative for session tokens.             │
│  Local staff row is authoritative for role and membership.      │
└─────────────────────────────────────────────────────────────────┘
```

| Field | Invited | Authenticated |
|---|---|---|
| `invite_code` | set | retained |
| `supabase_user_id` | NULL | set |
| `email` | set (required to reach this state) | set |
| Auth-gated features | blocked | permitted |

## Consequences

### Migration v3 additions to `staff` table

The following `ALTER TABLE` statements constitute database migration v3 in `lib/database.ts`. They must be applied inside the existing migration guard (version check before execution):

```sql
ALTER TABLE staff ADD COLUMN email TEXT;
ALTER TABLE staff ADD COLUMN role_level TEXT NOT NULL DEFAULT 'member';
ALTER TABLE staff ADD COLUMN invite_code TEXT;
ALTER TABLE staff ADD COLUMN supabase_user_id TEXT;
ALTER TABLE staff ADD COLUMN avatar_url TEXT;
```

All five columns are additive. Existing rows will receive `NULL` for nullable columns and `'member'` for `role_level` via the `DEFAULT`. The owner's row must be updated post-migration to set `role_level = 'owner'`.

### New files required

| File | Purpose |
|---|---|
| `lib/auth.ts` | Supabase OTP flow: send magic link, verify OTP, write `supabase_user_id` back to local `staff` row on success |
| `stores/auth.ts` | Zustand store: holds current authenticated user state (`supabase_user_id`, `role_level`, session token). Source of truth for auth state at runtime. |

### TypeScript type update

`StaffMember` in `types/index.ts` must be extended to reflect the new columns:

```ts
interface StaffMember {
  id: string;
  name: string;
  role: string;                   // free-text job title (unchanged)
  role_level: 'owner' | 'manager' | 'member';  // NEW
  color: string;
  voice_sample_uri?: string;
  avatar_initials: string;
  avatar_url?: string;            // NEW
  email?: string;                 // NEW
  invite_code?: string;           // NEW
  supabase_user_id?: string;      // NEW
}
```

### Security notes

- **`invite_code` is not a secret by itself.** It identifies the profile but does not authenticate the person. The OTP step over email is the authentication gate. Do not treat a bare invite code as proof of identity.
- **`supabase_user_id` must be validated server-side.** The local SQLite value is the cached result of a verified auth event. Any action with privilege implications must be re-validated against a live Supabase session token — not just the locally-stored UUID.
- **`invite_code` is permanent.** If a staff member is removed from the org, their `staff` row should be deleted (or `invite_code` rotated) to prevent re-authentication via a stale link.
- **Email is sensitive PII.** The `email` column must not appear in logs, transcript exports, or AI prompts. Sentinel must flag any code path that includes `email` in data sent to external services other than Supabase Auth.
- **Owner bootstrap:** The owner's `role_level = 'owner'` flag is the root of the permission hierarchy. There must be exactly one `owner` row per database. Enforce this constraint at the application layer — SQLite does not support a native `UNIQUE WHERE` partial index in expo-sqlite without a trigger.
- **`@supabase/supabase-js` is already in `package.json`** but was not previously integrated. `lib/auth.ts` will be its first use point. Boss's approval for this integration is recorded in this decision.
